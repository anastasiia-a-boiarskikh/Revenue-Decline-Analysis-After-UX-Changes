### Анализ снижения выручки после UX-изменений
Продукт — игра, кодовое название — Mines&Moles

- Сфера: game-аналитика
- Цель: сформулировать гипотезы о причинах снижения выручки и проанализировать изменения метрик, работая с деревом метрик продукта
- Результат: аналитический отчет с детальной информацией по метрикам, выводами и рекомендациями
- Стек: Python (pandas, matplotlib), JupyterLab
- Артефакты: аналитический отчет (*Аналитическая записка_Mines&Moles (1).pdf*), дерево метрик (в аналитическом отчете), JupyterNotebook с расчетами и визуализациями (*revenue_decline_analysis_after_ux_changes.ipynb*)

В аналитическом отчете собраны визуализации динамики различных метрик продукта (по дереву метрик) с промежуточными выводами; в конце приведены общие выводы по проекту и рекомендации

Требуемые для запуска библиотеки и версии:
- pandas==2.1.4
- matplotlib==3.8.0
- seaborn==0.13.2

#### Описание данных
1) Данные пользователей — таблица **users**:
- `device_id` (type: String) — идентификатор устройства пользователя, на которое установлена игра 
- `user_id`(type: Int) — идентификатор зарегистрированного в игре пользователя, NULL — если пользователь не зарегистрирован 
- `install_time` (type: DateTime) — дата и время установки приложения 
- `country_code` (type: String) — код страны пользователя (определяется по магазину приложений, в котором была скачана игра) 
- `platform` (type: String) — операционная система устройства, на которое установлена игра (iOS или Android) 
- `traffic_type` (type: String) — тип источника привлечения пользователя (органический, платный, реферальный) 
- `source` (type: String) — конкретный источник привлечения пользователя 
- `campaign` (type: String) — идентификатор рекламной кампании (только для платного трафика, для остальных — NULL) 

2) Данные о событиях — таблица **events**:
- `device_id` (type: String) — уникальный идентификатор устройства пользователя 
- `session_id` (type: Int) — идентификатор сессии пользователя 
- `time`(type: DateTime) — дата и время события 
- `event_name` (type: String) — название события, список событий:
> - *app_start* — старт приложения
> - *registration_complete* — прохождение регистрации
> - *game_launched* — запуск игры
> - *store_open* — открытие магазина
> - *item_add_to_cart* — добавление предмета в корзину
> - *checkout_load* — открытие чекаута
> - *order_completed* — завершение заказа

3) Данные о прохождении уровней — таблица **game_sessions**:
- `device_id` (type: String) — уникальный идентификатор устройства пользователя 
- `session_id` (type: Int) — идентификатор сессии пользователя 
- `level_id` (type: Int) — идентификатор уровня 
- `time_start` (type: DateTime) — дата и время начала прохождения уровня 
- `time_end` (type: DateTime) — дата и время окончания прохождения уровня 
- `success_fact` (type: Int) — завершение уровня пользователем: 1 — завершён, 0 — не завершён 
- `duration_sec` (type: Int) — длительность прохождения уровня в секундах 

4) Данные о транзакциях в игровом магазине — таблица **transactions**:
- `device_id` (type: String) — уникальный идентификатор устройства пользователя 
- `session_id` (type: Int) — идентификатор сессии пользователя 
- `transaction_id` (type: Int) — идентификатор транзакции 
- `time` (type: DateTime) — дата и время транзакции 
- `payment_type` (type: String) — способ оплаты (банковская карта, PayPal, внутриигровой кошелёк) 
- `item_name` (type: String) — название товара в магазине 
- `price` (type: Float) — цена товара в долларах 
- `status` (type: String) — результат транзакции (*success* — оплата прошла успешно, *failed* — оплата не прошла)

#### Ход проекта
1. Сформулированы гипотезы о причинах снижения выручки, наример:
   - Новый алгоритм генерации уровней предлагает слишком сложные уровни в начале. Пользователи не успевают пройти достаточно уровней, чтобы "залипнуть" и начать совершать покупки во внутриигровом магазине. По дереву метрик: снижается доля пользователей, у которых пройдено более 10 уровней → снижается конверсия в покупку → снижается средняя выручка на активного пользователя → снижается выручка;
   - Из-за редизайна внутриигрового магазина пользователи не могут найти нужные им предметы, поэтому тратят больше времени на поиск и покупают меньше. По дереву метрик: снижается кол-во предметов в заказе → снижается средний чек заказа по внутриигровом магазине → снижается средняя выручка на активного пользователя → снижается выручка;
   - другие гипотезы приведены в аналитическом отчете
2. Таблицы из схемы mines объединены и преобразованы (SQL) для получения основной таблицы для дальнейшей работы в Jupyter Notebook
3. В Jupyter Notebook рассчитаны и визуализированы следующие метрики (движемся по ветвям дерева метрик, ищем причины изменения верхнеуровневых метрик в динамике метрик более низкого уровня):
    - динамика MAU, в т.ч. с сегментацией по платформам
    - динамика кол-ва новых пользователей по источникам привлечения
    - возвращаемость (retention rate) пользователей по когортам
    - средняя доля пройденных уровней на игровую сессию по месяцам
    - ARPAU по месяцам
    - динамика конверсии воронки с заданным окном атрибуции по месяцам
    - средняя цена и кол-во предметов в заказе по месяцам
    - доля пользователей, у которых пройдено более 10 уровней, по месяцам
4. Динамика метрик проанализирована, анализ представлен в аналитическом отчете с промежуточными выводами, затем сделаны общие выводы по проекту, приведены рекомендации

#### Выводы и рекомендации по проекту
В июле 2024 года одновременно снизились доля пользователей, у которых пройдено более 10 уровней, и удержание (retention rate) пользователей. Первое повлияло на конверсию в покупку во внутриигровом магазине, что в свою очередь снизило среднюю выручку на активного пользователя. Второе снизило кол-во удержанных пользователей, что в свою очередь снизило общее кол-во активных пользователей.

Вероятно, это связано с изменением алгоритма, генерирующего уровни. Новые уровни стали сложнее, что как оттолкнуло начинающих игроков
(они не успевают пройти достаточно уровней, чтобы “залипнуть” в игру и начать совершать покупки во внутриигровом магазине), так и снизило
удержание пользователей (уменьшилось кол-во пройденных уровней в сессию, это ухудшило пользовательский опыт).

Рекомендации: можно вернуться к предыдущему алгоритму генерации уровней, либо изменить существующий, если изменение предыдущего алгоритма генерации необходимо
